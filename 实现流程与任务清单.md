# To Collect — 实现流程与任务清单

> 基于《项目需求.txt》与《uiux.pdf》，面向约百人级个人社交项目的开发规划。UI 以 **最新 Figma** 为准，本文档聚焦后端、数据与前后端联调。
>
> **Figma（最新）**：[Untitled](https://www.figma.com/design/YtxTXUolA8gYPX40uYMpLm/Untitled?node-id=0-1&t=3wKlq0MJCt7bUhhb-1) — 主页已改版，需展示**消息页面**，用户通过消息列表进入 Chatroom。

## 一、技术栈确认

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端 | React + JavaScript | 组件化、与 Figma UI 对接 |
| 后端/服务 | Supabase | 认证、数据库、存储、可选 Realtime |
| 部署 | DigitalOcean VPS | 托管前端构建产物 + 可选 Node 服务 |
| 版本控制 | GitHub | 代码托管，不用于部署 |
| 外部 API | ChatGPT Image 1.5 API | 头像生成、可选背景去除 |

## 三、Supabase 数据模型（核心表结构建议）

- **users_ext**（或用 Supabase Auth 的 user_metadata / 单独表）
  - 与 `auth.users` 关联：`id` = auth uid
  - `username`（唯一）、`avatar_url`、`created_at`、`updated_at`

- **objects**（平台所有物品，永久存档）
  - `id`、`owner_id`、`image_url`、`description`、`category`、`created_at`
  - 谁上传、是否在某人「当前 20 件」里，用 profile 表表达

- **user_profile_objects**（用户当前展示的 20 件）
  - `user_id`、`object_id`、`position`（1–20）、`updated_at`
  - 替换 = 改这条记录指向的 `object_id`，或先插新 object 再改 position

- **reposts**
  - `user_id`、`object_id`、`created_at`

- **blog_posts**（每日心情）
  - `user_id`、`date`（日期）、`object_ids`（JSON 或关联表，最多 5 个）

- **conversations** / **messages**
  - 会话：`user_a`、`user_b`、`updated_at`
  - 消息：`conversation_id`、`sender_id`、`content`、`object_id`（可选，代表自己的物品）、`created_at`

- **categories**
  - `id`、`name`、`slug`（如 cups, pens, utensils, lamps）

存储桶建议：`avatars`、`objects`（物品图）。RLS 控制：用户可写自己的头像/物品，所有人可读。

## 七、风险与注意点

- **API Key 安全**：ChatGPT Image 1.5、去背景等必须走服务端（Supabase Edge Function 或 VPS 上的 Node 服务），不要写进前端代码或 .env 并提交到 GitHub。
- **存储与 RLS**：上传前校验文件类型与大小；RLS 限制每人只能删/改自己的元数据，物品图建议只增不删（符合「永久存档」）。
- **百人规模**：Supabase 免费/低价档即可；若用 Realtime，注意连接数限制；VPS 单机静态站即可。

按上述流程从阶段 0 做到阶段 9，即可完成「To Collect」从登录、头像、物品、主页、**消息页**、心情到聊天室的完整闭环，并与最新 Figma UI 对接。
